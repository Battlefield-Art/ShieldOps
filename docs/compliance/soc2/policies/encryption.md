## 1. Purpose

This policy defines how ShieldOps protects the confidentiality and integrity of data using cryptographic controls. It supports SOC 2 criteria **CC6.1, CC6.7, C1.1, C1.2**.

## 2. Scope

All data classified Confidential or Restricted (see `policies/data-classification.md`), plus authentication credentials and session material regardless of classification.

## 3. In-Transit Encryption

- **External TLS.** All public endpoints (`api.shieldops.io`, `app.shieldops.io`, `docs.shieldops.io`, `status.shieldops.io`) require **TLS 1.3**. TLS 1.2 is permitted only for customer-requested legacy integrations, and only with forward-secret cipher suites (ECDHE).
- **No plaintext HTTP.** HTTP listeners redirect to HTTPS. HSTS is set to 1 year with `includeSubDomains; preload`.
- **Certificate issuance.** ACME via Let's Encrypt (default) or AWS Certificate Manager (ALBs). Certificates renew automatically at least 30 days before expiry.
- **Internal TLS.** Service-to-service traffic inside the cluster uses mTLS via the service mesh. Certificates rotate every 24 hours.
- **Database in transit.** PostgreSQL connections require TLS with `sslmode=verify-full`. Clients pin the RDS CA bundle.
- **Kafka and Redis.** TLS enforced with client authentication via mTLS or SCRAM.
- **Connectors outbound.** All outbound HTTPS calls use the system trust store. Certificate validation is never disabled.

## 4. At-Rest Encryption

- **Block storage (EBS, RDS volumes):** AES-256 via AWS KMS. KMS keys are customer-managed (CMKs), one per environment (dev, staging, prod).
- **Object storage (S3):** SSE-KMS with a dedicated CMK per bucket classification.
- **Database (PostgreSQL):** Transparent at-rest encryption via RDS. Application-level field encryption is additionally applied to Restricted columns.
- **Backups and snapshots:** Encrypted with the same CMK family as the source. Cross-region copies use a region-local CMK via grants.
- **Kafka and Redis at rest:** Encryption enabled on underlying volumes; message-level encryption for any topic carrying Restricted data.
- **Logs and metrics:** Encrypted at rest in the log store. Log ingestion strips credentials via `structlog` redaction before persistence.
- **Laptops:** Full-disk encryption (FileVault on macOS, BitLocker on Windows) enforced by MDM.

## 5. Application-Level Field Encryption

Restricted columns that are searchable via exact match (e.g., emails used for lookup) use deterministic encryption; columns that are purely stored (e.g., customer secrets, MFA seeds) use probabilistic AES-256-GCM with a random 96-bit nonce.

Implementation lives in `src/shieldops/compliance/field_encryption.py`. Keys are fetched from AWS KMS on application start and cached in memory only. The plaintext key never touches disk.

## 6. Password and Secret Hashing

- **User passwords:** Argon2id, parameters `memory=64MB`, `iterations=3`, `parallelism=4`. Hashes stored in `users.password_hash`.
- **API keys:** SHA-256 of a 32-byte random value, stored in `api_keys.key_hash`. The plaintext key is shown to the user exactly once at creation time.
- **Session tokens:** JWT signed with HS256; signing key held in AWS Secrets Manager and rotated quarterly.

## 7. Key Management

### 7.1 Inventory

All cryptographic keys (symmetric keys, asymmetric key pairs, signing keys, TLS private keys) are inventoried in `docs/compliance/soc2/key-inventory.yaml` (source of truth: AWS KMS + Secrets Manager + 1Password shared vault).

Each inventory entry records: purpose, algorithm, length, owner, location, creation date, rotation schedule, destruction date (if applicable).

### 7.2 Generation

- Generated via cryptographically secure RNGs (AWS KMS, OpenSSL, Python `secrets`).
- Keys must never be generated by LLMs, spreadsheets, or hand.

### 7.3 Storage

- Production keys: AWS KMS (preferred) or AWS Secrets Manager.
- Break-glass credentials: 1Password shared vault `prod-break-glass`, gated by biometric + second approver.
- Developer keys: never stored in git, CI variables for non-prod only. Production CI/CD pulls secrets from AWS Secrets Manager via OIDC-federated GitHub runners; no static credentials in GitHub Actions.

### 7.4 Rotation

| Key type                                   | Rotation cadence | Trigger for immediate rotation                     |
|--------------------------------------------|:----------------:|----------------------------------------------------|
| TLS certificates                           | Automatic (ACM)  | Suspected private key exposure                     |
| JWT signing key                            | Quarterly        | Suspected compromise, departure of key holder      |
| Database master credentials                | Quarterly        | Role change, suspected compromise                  |
| KMS customer-managed keys                  | Annual (auto)    | Suspected compromise                                |
| Customer-facing API keys                   | Max 365 days     | User request, revocation via dashboard             |
| Service account credentials                | 90 days          | Suspected compromise                                |
| Break-glass credentials                    | Quarterly        | Any use, any key-holder departure                   |

See `procedures/secrets-rotation.md` for the operational procedure.

### 7.5 Destruction

- AWS KMS: schedule key deletion with the minimum 7-day waiting period; require a secondary approver.
- Secrets Manager: delete with 7-day recovery window.
- 1Password items: move to archive; purge after 90 days.
- Physical media: N/A (ShieldOps does not handle HSMs on-premise).

## 8. Cryptographic Agility

- No use of MD5, SHA-1, DES, 3DES, RC4, or export-grade ciphers anywhere.
- `bandit` scans in CI fail the build on detected weak crypto.
- Post-quantum readiness tracked in `docs/security/pqc-readiness.md`; ShieldOps will migrate to NIST PQC standards within 24 months of finalization for all externally visible TLS and signing.

## 9. Customer-Managed Keys (BYOK)

Enterprise customers may bring their own KMS keys. Implementation:

- A customer KMS key (in the customer's AWS account) is granted key usage permission to the ShieldOps production role via a cross-account grant.
- ShieldOps uses the key for envelope encryption of that customer's Restricted data.
- Customer can revoke the grant at any time; revocation causes immediate loss of access to their encrypted data (documented in the BYOK addendum).
